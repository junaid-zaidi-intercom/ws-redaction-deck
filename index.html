Background

Why redaction is required

Context — HSG / BPO Separation

BPOs are treated as separate entities — their data is PII to others

BPO (Business Process Outsourcing) agents operate under a strict identity boundary. A BPO agent's real name and photo are PII from the perspective of the brand's workspace — other teammates should not be able to see BPO agent identity. This is a data separation requirement between two distinct organisations sharing a platform.

The Risk

BPO agent PII visible to brand teammates

Events like admin_is_typing carry adminName and adminAvatar in the payload — exposing real BPO agent identity to teammates in the brand's inbox.

The Problem

No separate WebSocket channel

Inbox (teammates) and the SDK (end users) receive the exact same event. Redacting PII for teammates means end users are also redacted — there is no way to broadcast differently to each consumer.

Technical Detail

Payload: before vs. after redaction

FF OFF
{
  "admin_id": "12345",
  "adminName": "Jane Smith",
    // ⚠ real PII
  "adminAvatar": "https://...",
    // ⚠ real PII
  "hasDefaultAvatar": false
    // no isRedacted field
}

Real PII embedded. No isRedacted field present.

FF ON — v1 (broke SDK)
{
  "admin_id": "12345",
    // adminName absent
    // adminAvatar absent
    // isRedacted absent



}

Fields stripped entirely. SDK had no fallback — typing indicator broke.

FF ON — now (fixed)
{
  "admin_id": "12345",
  "adminName": "Admin",
    // safe placeholder
  "adminAvatar": "<default>",
    // safe placeholder
  "hasDefaultAvatar": true,
  "isRedacted": true
}

Placeholders keep SDK alive. isRedacted: true signals to resolve via admin_id.

How Inbox handles this already

Inbox ignores adminName / adminAvatar entirely when the FF is on. It looks up the teammate record by admin_id from its own in-memory store — this is the pattern we want the SDK to follow for the public-facing display name and avatar.

History

What we shipped and what broke

1

Initial FF rollout Inbox ✓

When hsg-bpo-separation-websocket-redaction is ON, we stopped embedding adminName / adminAvatar in the payload. Inbox resolves the admin identity from admin_id in the event — so inbox kept working perfectly.

2

SDK broke SDK ✗

The SDK was reading adminName and adminAvatar directly from the event payload. With those fields absent, the "typing" indicator showed blank / broken UI in customer-facing surfaces. We rolled the FF back.

3

Interim fix — safe defaults injected

When the FF is ON, we now send safe placeholder values so nothing crashes: adminName = "Admin", adminAvatar = default avatar, isRedacted = true. This keeps the SDK functional while we align on the correct long-term approach.

4

Today Needs SDK work

We want to re-enable the FF properly. For that, the SDK must resolve the agent's public-facing display name and avatar from the admin_id in the payload — not from the stripped PII fields.

Ask

What we need from SDK implementers

Important

When the FF is ON, isRedacted is always true — there is no per-event mixed state. The FF is the switch; all events from that point forward are redacted.

1

Stop relying on adminName / adminAvatar from the payload

Once the FF is on, these fields will always contain safe placeholders ("Admin" / default avatar), not real identity data. Reading them will never give you the agent's actual name or photo.

2

Resolve display identity via admin_id

Use the admin_id in the event to look up the agent's customer-facing display name and avatar from your own data source (teammate store, REST API, etc.). This is the correct identity to show end users.

3

Graceful fallback if resolution fails

If admin_id cannot be resolved (e.g. not yet in cache), fall back gracefully to a generic "Support Agent" label and the default avatar — the same values already in the placeholder fields. Do not show a broken or empty UI.

Summary

Next Steps

We will

Re-enable the FF

Once SDK changes are deployed, we'll turn on hsg-bpo-separation-websocket-redaction. From that point, all admin_is_typing events will have real PII stripped — always.

SDK will

Resolve identity via admin_id

Stop reading adminName / adminAvatar from the payload. Look up the agent's public display info using admin_id. Fall back to "Support Agent" + default avatar if unresolved.

Coordination required

The FF must only be re-enabled after the SDK changes are live. There is no per-event toggle — it is all-or-nothing. Turning on the FF before the SDK is updated will break the typing indicator again.

Why this matters

The primary goal is preventing brand teammates from seeing BPO agent identity in inbox. Because there is no separate WebSocket channel, end users via the SDK are also redacted as a side effect — which is why the SDK needs to resolve agent display info via admin_id rather than reading it from the payload.
